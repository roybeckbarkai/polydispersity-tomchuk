# MATLAB vs Python Comparison Report

This report compares the MATLAB reference implementation in `matlab-tenorSAXS/` against the Python implementation in the main application directory. The comparison is split into:

1. SAXS signal creation
2. TENOR-SAXS analysis

The focus is on functional correspondence, algorithmic differences, hard-coded constants ("magic numbers"), and the practical effects those differences are expected to have.

## Scope

MATLAB files reviewed:

- `matlab-tenorSAXS/Scatter2D.m`
- `matlab-tenorSAXS/size_distribution_disc.m`
- `matlab-tenorSAXS/MG_extract.m`
- `matlab-tenorSAXS/fit_I_r_theta_ratios_weighted_centered.m`
- `matlab-tenorSAXS/best_origin_quad_b_fast_bins.m`

Python files reviewed:

- `sim_utils.py`
- `tenor_saxs.py`
- `analysis_utils.py`

## 1. SAXS Signal Creation

| Step | MATLAB function(s) | Python function(s) | Difference in functionality | Magic numbers / hard-coded rules | Expected differences in outputs |
|---|---|---|---|---|---|
| Distribution PDF construction | `Scatter2D.m` internal distribution setup; `size_distribution_discrete_no_max` in `size_distribution_disc.m` | `get_distribution`, `_distribution_pdf_on_grid` in `sim_utils.py` | MATLAB and Python support the same main family types, but MATLAB's discrete path is treated as part of the reference simulator logic, while Python separates continuous PDF evaluation from later discrete ensemble construction. Python is cleaner structurally, but less tightly coupled to the original simulation flow. | MATLAB discrete routine uses `threshold`, `xmin`, `weight_power`, dense `10000`-point grid, and mean normalized to `1`. Small discrepancies can appear near truncation, especially for narrow or highly skewed distributions. |
| Discrete ensemble generation | `size_distribution_discrete_no_max` | `sample_size_distribution`, `_build_discrete_distribution_from_pdf` | MATLAB partitions a weighted fine-grid CDF and then applies a final affine mean/variance correction to enforce target moments. Python now mirrors the weighted-CDF plus center-of-mass binning idea, but does not apply the final exact affine correction step. | MATLAB uses `10000` grid points, `threshold` typically near `0.999`, `xmin`, and `weight_power`. Python uses `threshold=0.999`, minimum `ensemble_members=3`, fallback if fewer than 2 members survive, and `weight_power=0.0` in the discrete sampler unless otherwise handled upstream. | Python discrete ensembles should now be much more stable than before, but may still miss the target mean/relative variance slightly compared with MATLAB, especially for coarse ensembles or hard truncation near `xmin`. |
| Mean-size variable definition | `Scatter2D.m` | `run_simulation_core` | MATLAB uses `rg` as primary input and internally converts to particle-size-like quantities depending on the form factor branch. Python uses string-based `form_factor_model` and fixed conversions such as `sqrt(5/3)`. | Python is clearer and safer. MATLAB is more compact but easier to misuse because geometry is encoded in curvature constants. |
| Form-factor model dispatch | `Scatter2D.m` | `get_form_factor_kernel`, `sphere_form_factor`, `debye_form_factor`, `shell_form_factor`, `thin_rod_form_factor`, `thin_disk_form_factor`, `guinier_curvature_form_factor` | MATLAB dispatches partly through special `Nu` values and otherwise falls back to a Guinier-curvature exponential. Python uses explicit model names and default `phi2` values. | Python avoids accidental model switching from near-matching numeric `Nu`, while MATLAB may reproduce the original paper workflow more literally. |
| Higher-order curvature handling | `Scatter2D.m` exponential with `Nu` and optional higher-order term | `guinier_curvature_form_factor` with `phi2`, `phi3` | MATLAB primarily exposes a quartic correction through `Nu`, with some special-case exact forms. Python generalizes to both `phi2` and `phi3`, which is more flexible for experimentation. | MATLAB hard-codes coefficients in the exponent and special-case branches. Python defaults `phi2` to sphere or chain values and `phi3=0.0`. | Python can represent more models, but if the MATLAB workflow relies on special geometric overrides, direct equivalence is not guaranteed unless the same explicit model is selected. |
| Scattering-strength weighting of ensemble members | `Scatter2D.m` multiplies by `r_vect(ii).^Weight_power .* p(ii)` | `run_simulation_core` multiplies kernel matrix by `r_vals**weight_power` | Both codes support weighted averaging, but MATLAB exposes this as an integral part of the signal model, while Python derives a default from the selected form factor and can override it with `weight_power`. | MATLAB commonly uses `Weight_power=4` in discrete distribution generation and intensity weighting logic. Python defaults depend on model: sphere `0.0`, shell `4.0`, rod `2.0`, disk `4.0`, unless explicitly overridden. | Differences will appear when the same distribution is used but different effective weighting assumptions are active. This matters strongly for shell/rod/disk families and for any calibration based on weighted moments. |
| 1D intensity generation | `Scatter2D.m` computes intensity directly over the 2D detector grid | `run_simulation_core` computes `i_1d_ideal` on `q_1d` first, then interpolates onto the 2D detector | MATLAB evaluates on the detector coordinates directly. Python first builds a radial 1D curve and then interpolates it to the 2D grid. | Python uses `q_1d = logspace(log10(1e-3), log10(detector_q_max*1.5), q_steps)` with default `q_steps=200`. MATLAB uses the detector grid natively. | Python may introduce small interpolation artifacts or slight center-pixel differences, especially for sharp curvature, very high smearing, or coarse `q_steps`. MATLAB is more direct in 2D. |
| Detector geometry to q-grid | `Scatter2D.m` | `get_detector_q_max`, `build_detector_q_grid` | MATLAB uses detector half-side, sample distance, wavelength, and odd-sized pixel layout directly. Python uses either `q_max` or instrument geometry helpers and builds a symmetric grid with `np.linspace`. | MATLAB defaults include `SD_dist=150`, `lambda=0.15`, `det_hside=3.5`, `DETpix` interpreted through `Npix=round(DETpix/2)`. Python defaults include `sample_detector_distance_cm=150.0`, `wavelength_nm=0.15`, detector side fallback from `pixel_size_um`. | Small q-grid alignment differences are expected, especially because MATLAB naturally centers an odd detector and Python often operates on even image sizes from the app. |
| Detector centering convention | `Scatter2D.m` | `build_detector_q_grid`, `run_simulation_core` | MATLAB centers around an odd grid (`2*Npix+1`), which gives a true central beam pixel. Python allows even grids and explicitly treats `(pixels//2, pixels//2)` as the center proxy. | Python comment documents this convention. MATLAB assumes exact center symmetry. | Flux normalization and the forward-most intensity can differ slightly for even detector sizes in Python. This can affect low-q noise and apparent Guinier extraction. |
| Smearing / PSF on the simulated 2D image | `Scatter2D.m` with `filter2(PSF0,F,"same")` | `gaussian_filter` in `run_simulation_core` | MATLAB applies an arbitrary supplied PSF kernel. Python currently applies separable Gaussian smoothing parameterized by pixel sigmas `smearing_x`, `smearing_y`. | MATLAB PSF is externally supplied and can be non-Gaussian. Python assumes Gaussian blur only. | MATLAB can model detector blur or anisotropy more generally. Python is simpler but may not reproduce asymmetric or non-Gaussian instrumental broadening exactly. |
| Noise model | `Scatter2D.m` | `run_simulation_core` | MATLAB does not use Poisson sampling directly when `Nois<0`; instead it rescales to a target center intensity and adds Gaussian noise with variance proportional to intensity. Python uses actual Poisson draws when `noise=True`. | MATLAB negative-noise branch uses `sqrt(F).*randn`, peak normalization, and no exact Poisson. Python uses `np.random.poisson`. | Python should be more physically faithful in photon-counting regimes. MATLAB may produce smoother behavior at moderate counts, but less accurate tails under very low flux. |
| Flux normalization target | `Scatter2D.m` | `run_simulation_core` | MATLAB scales by detector-maximum-like conventions in the noisy branch. Python defines `flux` as the expected photons in the nearest-to-center pixel after smearing and before Poisson noise. | Python explicitly uses `cy = pixels//2`, `cx = pixels//2`. MATLAB uses internal peak scaling in the generated image. | Reported flux values are not directly interchangeable. Low-flux comparisons can differ even if nominal numbers look similar. |
| Radial reduction of 2D image | MATLAB workflow mostly keeps the 2D image in `Scatter2D.m`; later radial logic is separate | `radial_average_detector_image` | Python has a shared, reusable 2D-to-1D reduction function used by both simulation and analysis. MATLAB keeps the simulation image native and performs specific reductions elsewhere. | Python uses linear or logarithmic binning, `n_bins`, and validity filtering. | Python is internally more consistent across simulation and analysis, but not a literal line-by-line port of the MATLAB simulation path. |
| Guinier-validity masking during simulation | `Scatter2D.m` sets parts of the exponent to `nan` when the quartic term dominates | No direct equivalent in `run_simulation_core` | MATLAB contains an explicit guard against using the low-order expansion outside its valid regime. Python does not apply an analogous `nan` mask in the simulator. | MATLAB rule: `abs(-(s2/3)q^2) < abs((0.5*nu*s4)q^4)` marks invalid region. Python has no exact analog. | MATLAB may suppress unphysical high-q tail behavior in approximate models more aggressively. Python may carry the approximate form factor farther into q unless an exact model is selected. |
| Detector kernel housekeeping | `makeKernelOddCentered`, `gaussian_discrete`, `bartlett2d`, `tri1d` in `Scatter2D.m` | No direct equivalent | MATLAB includes helper routines for centering odd kernels and constructing discrete smoothing kernels. Python delegates to SciPy Gaussian filtering and does not expose equivalent Bartlett/triangular helpers. | Multiple helper-specific default modes and kernel-size assumptions in MATLAB. | MATLAB gives more manual control over PSF construction. Python is more compact but less feature-complete for custom detector kernels. |

## 2. TENOR-SAXS Analysis

|---|---|---|---|---|---|
| Apparent Guinier radius from 2D data | `MG_extract` calling `best_origin_quad_b_fast_bins` | `apparent_rg_from_2d`, `estimate_guinier_parameters` | MATLAB estimates apparent `Rg^2` from `log(I)` vs `q^2` using a window-search quadratic-origin method with bootstrap confidence logic. Python radially averages the 2D image first and then performs iterative linear Guinier fitting on the 1D curve. | MATLAB uses `nBins=200`, `B=20`, `threshold=-1/6`, `min_pts=6`, tolerance settings in `best_origin_quad_b_fast_bins`. Python uses up to 5 iterative fits, `limit=0.9` for spheres, `min_pts=6`. | Python is simpler and faster, but less robust to subtle curvature, anisotropy leakage, and noisy low-q regions. MATLAB should be more conservative in choosing the valid Guinier window. |
| PSF pair generation for TENOR extraction | `MG_extract` default `Pxn` construction | `build_default_psf_pairs` | MATLAB builds specific anisotropic Gaussian pixel-width pairs, including defaults resembling one narrow and one broader orthogonal blur. Python scans a configurable family of PSF quartets generated from start values, step size, and secondary ratio. | MATLAB hard-coded starts include values like `(5,1)` and `(1,9)` through internal `Pxn` logic. Python defaults are `sigma_x_start=1.2`, `sigma_y_start=0.6`, `sigma_step=0.4`, `pair_count=5`, `secondary_ratio=0.5`. | Python explores a smoother parameter family and is easier to tune, but it does not replicate MATLAB’s exact default quartets, so the selected best pair can differ substantially. |
| Digital smearing of the observed image | `MG_extract` uses `filter2` with Gaussian PSFs | `apply_anisotropic_gaussian`, `extract_pair_observables` | Both blur the same observed image with two anisotropic Gaussian filters and analyze the log-ratio. | Log-ratio map construction | `MG_extract` | `extract_pair_observables` | Both compute a log-ratio between two digitally smeared images. Python clamps with `np.maximum(...,1e-12)` before taking logs; MATLAB relies more on filter output positivity and selected fit mask. | Python hard floor `1e-12`. MATLAB has no obvious identical floor in the visible code path. | Python is safer numerically under severe noise or sparse counts, but the floor can bias the weakest pixels slightly. |
| Fit-domain selection in q | `MG_extract` | `extract_pair_observables` | MATLAB repeatedly overwrites `qrng` and ends with a tight upper limit around `0.79/sqrt(R_g_nofilt)` plus dead-pixel trimming. Python uses `qrg_limit=0.85` by default and a dead-zone based on `psf_truncate * sigma`. | MATLAB sequence includes `1.4765`, `1.35`, `0.9`, then final `0.79`. Python uses `qrg_limit=0.85`, `psf_truncate=4.0`. | Python usually fits a slightly wider q-range than MATLAB, which may improve information use in clean data but hurt stability when higher-order terms or noise dominate. |
| Dead-pixel exclusion | `MG_extract` | `extract_pair_observables` | Both exclude a central region affected by PSF overlap. MATLAB uses `deadpix=2*max(Pxn)`. Python converts sigma values to an excluded q radius using `2*ceil(psf_truncate*sigma)`. | MATLAB simple `2*max(Pxn)`. Python `psf_truncate=4.0` and `ceil`. | Excluded central region sizes will not match exactly, which shifts the low-q information entering the fit. |
| Weighted polynomial fit for `G(q)` and `M(q)` | `fit_I_r_theta_ratios_weighted_centered` | `fit_weighted_centered_tenor_model` | MATLAB uses QR-based weighted least squares with SVD fallback if rank deficient. Python currently defaults in the function are `use_m3=True`, `use_g3=True`, though your stability configuration for large runs was changed downstream to false/false. Python ridge default `1e-6 * trace(X^T W X)`. | MATLAB should be more numerically faithful and less sensitive to conditioning if rank drops. Python is more configurable and now more stable with ridge, but coefficient values can differ, especially under high Poisson noise. |
| Basis definition for the anisotropic term | `fit_I_r_theta_ratios_weighted_centered` | `fit_weighted_centered_tenor_model` | MATLAB uses `M` columns built from `r*cos(2th)`, `r^2*cos(2th)`, optionally `r^3*cos(2th)`, where `r` is already the squared radial variable passed in by `MG_extract`. Python uses the same effective basis via `q_sq*cos2`, `(q_sq**2)*cos2`, `(q_sq**3)*cos2`. | Both rely on implicit interpretation of `r` as `q^2`. | Centering transform for `G` coefficients | `fit_I_r_theta_ratios_weighted_centered` | `fit_weighted_centered_tenor_model` | Both fit in a centered basis and map back to the original polynomial coefficients. | Weighting scheme in the fit | `fit_I_r_theta_ratios_weighted_centered` via `weight` input from `MG_extract` | `fit_weighted_centered_tenor_model` via `intensity_weights` from `extract_pair_observables` | MATLAB uses `double(I_mat + sqrt(I_mat))`-style weights on the fit mask. ## Bottom-Line Summary

### SAXS signal creation

- The Python simulator is more modular, easier to configure, and clearer about geometry and form-factor selection.
- The MATLAB simulator is more "directly physical" in some areas because it computes on the 2D detector grid itself and allows arbitrary PSF kernels.
- The biggest remaining simulator mismatch is the discrete ensemble exact-moment correction present in MATLAB but not yet fully mirrored in Python.
- Another important mismatch is the noise model: Python uses Poisson noise, while the reviewed MATLAB branch uses a Gaussian approximation tied to intensity.

### TENOR-SAXS analysis

- The core polynomial extraction idea is shared between MATLAB and Python.
- The Python implementation is not a strict port. It adds several layers intended for automation and stability: multi-pair candidate scanning, plausibility filtering, reconstruction-based validation, explicit simulation calibration, and optional apparent-`Rg` matching.
- MATLAB remains closer to the original extraction logic at the function level, especially in apparent `Rg` estimation and in the QR/SVD-based weighted fit.
- Python is better suited for large automated benchmark studies, but it will not exactly reproduce MATLAB numbers unless the defaults, fit window, weighting rule, solver, PSF definitions, and calibration logic are aligned more tightly.

## Most Important Expected Sources of MATLAB vs Python Disagreement

1. Apparent `Rg` estimation:
   MATLAB uses a more specialized low-q selection method; Python uses radial averaging plus iterative Guinier fitting.
2. PSF default definitions:
   MATLAB default blur pairs and Python default blur scan are not the same.
3. Fitting solver:
   MATLAB uses QR/SVD weighted least squares; Python uses ridge-regularized normal equations.
4. Weighting in the fit:
   MATLAB uses `I + sqrt(I)`-type weights; Python uses mostly raw intensity weights.
5. Signal-generation path:
   MATLAB simulates directly on the 2D detector grid; Python simulates a 1D curve and interpolates to 2D.
6. Noise model:
   MATLAB reviewed code uses Gaussian noise approximation; Python uses Poisson noise.
7. Discrete ensemble construction:
   Python now tracks MATLAB much more closely, but does not yet include the final affine exact-moment correction.

## Recommended Next Alignment Steps

If the goal is to make the Python implementation track the MATLAB reference more closely, the highest-value next changes would be:

1. Port the final affine exact-moment correction from `size_distribution_discrete_no_max`.
2. Add an optional QR/SVD solver path to `fit_weighted_centered_tenor_model`.
3. Add an optional MATLAB-like `I + sqrt(I)` weighting rule.
4. Add an optional MATLAB-like apparent-`Rg` estimator modeled on `best_origin_quad_b_fast_bins`.
5. Add a "MATLAB reference defaults" PSF-pair preset so both implementations start from the same blur quartet family.
